## 1. 背景与目标

大语言模型推理的瓶颈在于显存带宽——权重与激活的搬运远比算术操作昂贵。量化通过降低位宽直接压缩数据体积，但必须在精度和速度之间折衷。NVIDIA TensorRT‑LLM 没有一刀切，而是同时提供 FP4、FP8、INT4、SmoothQuant 等多种方案，借助 CUTLASS 模板库和手写 CUDA 核将它们做成 TensorRT 插件。

本文聚焦这些量化 GEMM 在 GPU 上的**数据布局、解量化过程和缩放因子管理**，解释它们如何集成到推理引擎。所有分析基于开源仓库，代码片段均标注文件与行号。

## 2. 核心设计与实现

### 2.1 FP4 Block‑Scaling：Block=16 的极致低位宽

`fp4GemmPlugin` 用 E2M1 格式（符号 + 2 指数 + 1 尾数，范围 [-6, 6]）。为补偿有限动态范围，权重和激活都采用 block‑level 缩放：每 16 个元素共用一个 8 位缩放因子（FP8 E4M3 存储）。缩放因子布局严格固定以匹配 Tensor Core 访问模式：权重 B 的 scale 按 `(K/4, N/16)` 排列；激活 A 的 scale 按 `(M/16, K/4)` 排列，M 维度按 128 对齐。

对齐策略在 `fp4GemmPlugin.cpp:48-52` 显式算出：

```cpp
int const M_padded = ((m + 127) / 128) * 128;
int const K_padded_scale_rows = ((k + 3) / 4 + k_pad_needed) * 4;
```

这种 padding 让每个 scale 行正好 128 字节对齐，GPU 可用单条 `ld.global.v4.f32` 加载一整行，避免 bank conflict。CUTLASS kernel 在主循环中执行解量化乘法：每个 block 从 16 个 4‑bit 值恢复浮点后乘以 scale，再走 Tensor Core mma；`global_scale` 在写回前做最终缩放，应对 E2M1 范围不足的情况。

### 2.2 FP8 Rowwise：每行的 Per‑Token 与 Per‑Channel 缩放

FP8 GEMM 插件 `fp8RowwiseGemmPlugin` 使用 Hopper 架构的 FP8 Tensor Core。激活和权重都是 1‑字节 E4M3 数据，但每行（token / channel）各自携带一个 32‑位浮点缩放因子。数据布局为：

- 激活 `A`：[M, K] 的 FP8，附带 [M] 的 `scale_token`。
- 权重 `B`：[K, N] 的 FP8，附带 [N] 的 `scale_channel`。
- 输出 `D`：[M, N]，可以是 FP16/BF16/FP32。

缩放因子在 `enqueue()` 中通过参数传入 CUTLASS runner：

```cpp
// fp8RowwiseGemmPlugin.cpp:enqueue 调用
mGemmRunner->gemm(outputs[0], inputs[0], inputs[1], nullptr, mQuantMode, m, n, k,
    reinterpret_cast<float const*>(inputs[2]),      // scale_token
    reinterpret_cast<float const*>(inputs[3]),      // scale_channel
    *bestTactic,
    reinterpret_cast<char*>(workspace), wsSize, stream);
```

CUTLASS kernel 在最终写回之前，将累加器上的结果乘以 `scale_token[thread_idx_m] * scale_channel[thread_idx_n]`。这一融合避免了单独的缩放核启动和显存往返，并且 scale 的广播完全在寄存器中完成。

### 2.3 INT4 Weight‑Only Groupwise：Packed 布局与激活预缩放

`weightOnlyGroupwiseQuantMatmulPlugin` 支持纯 INT4 权重的分组量化。权重的 physical layout 为每两个 4‑bit 元素打包成一个 8‑bit 字节，按 `(N, K/2)` 的 int8 张量存放，CUTLASS 将其直接 reinterpret 为 `cutlass::uint4b_t` 消费，无需运行时解包。每一组（group_size = 128 或 64）额外存储 per‑group 的 scale 和 zero，以 half 或 float 精度保存。

插件还支持 **Pre‑Quant Scale**——在 GEMM 之前对激活应用 per‑channel 缩放因子（类似 AWQ 的均衡思想）。如果使用 CUTLASS kernel，该缩放被抽到临时 workspace 中显式完成：

```cpp
// weightOnlyGroupwiseQuantMatmulPlugin.cpp:148-175
if (use_pre_quant_scale && !use_cuda_kernel) {
    act_ptr = reinterpret_cast<half const*>(workspace);
    pre_quant_scale_for_act<half>(m, k, mQuantAlgo,
                                  mPreQuantScaleInputIdx, inputs,
                                  workspace, stream);
}
```

对于 M < 16 的超小 batch，插件会选择手写 CUDA 核而非 CUTLASS，以降低 launch 开销。两个路径共享完全相同的量化参数和 packed weight 布局，上层逻辑通过 `checkTactic` 在 profiling 时决定。

### 2.4 SmoothQuant：W8A8 的 Token/Channel 缩放融合

SmoothQuant 把激活的量化难度迁移到权重：离线算出 smooth factor，使 `activation_scaled = activation * factor` 更易量化，`weight_scaled = weight / factor` 仍保精度。`smoothQuantGemmPlugin` 用 INT8 Tensor Core 做 W8A8 GEMM，并在 kernel 内部直接融合去量化缩放。

```cpp
// smoothQuantGemmPlugin.cpp: enqueue 中对小 batch 的自定义核
if (m <= 4) {
    tensorrt_llm::kernels::smooth_quant::Params params(…);
    tensorrt_llm::kernels::smooth_quant::int8_sq_launcher<half>(params, stream);
} else {
    m_sqGemmRunner->gemm(…, scale_token, scale_channel, …);  // 大 batch 走 CUTLASS
}
```

CUTLASS 的 epilogue 阶段把累加器结果乘以 `scale_token[m_idx] * scale_channel[n_idx]`，无需额外内存写入。

### 2.5 融合 Norm+Quant：从归一化直接到量化输出

`layernormQuantizationPlugin` / `rmsnormQuantizationPlugin` 把 LayerNorm/RMSNorm 与后续量化合并为单一 kernel，省去中间 FP16/FP32 的显存临时 buffer：

```cpp
// layernormQuantizationPlugin.cpp: dispatchDataType 内
invokeGeneralLayerNorm(
    …, reinterpret_cast<float const*>(scale),
    reinterpret_cast<float*>(dynamic_scale), …,
    reinterpret_cast<QuantT*>(normed_output_quant));
```

kernel 在完成均值/方差和归一化后立即 `round(clip(x * scale / max))` 写入 INT8/FP8 输出，同时把 per‑token 的 `max` 作为 `dynamic_scale` 给下游 GEMM。这种方式消除两次显存往返，批量推理吞吐提升明显。

## 3. 优化技术细节

### 3.1 内存对齐与向量化加载

所有量化 Plugin 在 profiling 阶段就计算好 padded 尺寸并分配 workspace。以 FP4 为例，A/B 的 scale 数组均 pad 到 128 字节对齐，满足 global memory 合并访问。CUTLASS 内部用 `cp.async` 或 TMA（Hopper）以 128/256 位粒度搬运，padding 逻辑在 `fp4GemmPlugin.cpp:48‑52` 显式体现，避免 kernel 内地址计算。

### 3.2 Tactic 缓存与动态 Kernel 选择

每个量化 GEMM Plugin 在构建期枚举 CUTLASS 所有 tile 配置并实际计时，缓存最快的 tactic。`weightOnlyGroupwiseQuantMatmulPlugin` 还引入 CUDA vs CUTLASS 开关：M < 16 时强制使用手写核（极小 batch 下 CUTLASS 的 wave 平铺开销不可忽略）。该逻辑在 `checkTactic` 和 `runTactic`（weightOnlyGroupwiseQuantMatmulPlugin.cpp:288‑302）执行。

### 3.3 Workspace 的线性分配与复用

插件通过 `nextWorkspacePtr` 顺序分配临时 buffer，所有临时张量共用一块连续空间，避免多次 `cudaMalloc`。`getWorkspaceSize` 一次性返回所需空间，实现零动态分配和确定性内存占用。

### 3.4 FP4 内联量化 kernel

`fusedGatedRMSNormQuant.cu` 实现 `cvt_float_to_fp4_inline`，在 warp 内通过 shuffle 找出 16 个元素最大值、计算 FP8 scale、将 float 转换为 4‑bit 并打包成 `uint32_t`。整个过程全部在寄存器中完成，与 RMSNorm 共享数据加载，达 O(N) 时间且仅一次 HBM 读取：

```cuda
// fusedGatedRMSNormQuant.cu: cvt_float_to_fp4_inline 核心逻辑
float sfValue = sfScaleVal * (localMax * reciprocal_approximate_ftz(FP4_E2M1_MAX));
__nv_fp8_e4m3 sfFp8 = __nv_fp8_e4m3(sfValue);
// 随后: 写入 sfFp8 到 scale 缓冲区,调用 fp32_vec_to_e2m1 打包
```

## 4. 性能与权衡

每种量化方案都是一组精确的 trade‑off：

- **FP4** 权重内存最低（4‑bit），但 block‑scale 增加存储、E2M1 范围窄需 global scale 兜底，解量化乘法在低延迟场景可能成瓶颈。
- **FP8** 借助 Hopper Tensor Core 拿到原生高吞吐，但每行/列 scale 增加内存开销与对齐约束，scale 动态范围受 float 精度限制。
- **INT4 weight‑only** 仅压权重、激活保持高精度，精度损失小；Gemm 内需从 packed 格式解包（或 `uint4b_t` 间接访问），group‑wise scale/zero 存储按 group_size 分摊。
- **SmoothQuant** 把激活量化难度迁到权重，使 W8A8 接近 FP16 精度，但需离线算 smooth factor，且 epilogue 中的 scale 乘法对 kernel 实现要求高。
- **融合 norm+quant** 极大减少显存带宽，但代码复杂度显著上升，需维护多种类型组合（float/half + int8/fp8）的模板实例化。

TensorRT‑LLM 通过 plugin 架构和 profiling，让用户根据模型特性、硬件代际和部署目标自由组合，而不是强制单一方案。

## 6. 总结

TensorRT‑LLM 的量化 GEMM 实现远不是简单的 `cudaMalloc` + `cublasGemmEx`。从 FP4 的 block‑scaling 对齐到 FP8 的 rowwise 缩放融合，再到 INT4 的 packed weight 与 SmoothQuant 的 scale 乘积，每一种方案都在数据布局和解量化流程上做了深度定制。这些操作通过 TensorRT plugin 形式无缝嵌入引擎构建，在 profiling 阶段自动选择最优 kernel，推理时仅通过少量指针传递即可生效。理解这些底层细节，有助于工程师更精准地评估不同量化路径对自身模型和硬件的收益与代价。
