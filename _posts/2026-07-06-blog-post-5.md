## 1. 背景与目标

随着模型参数规模向万亿级迈进，单 GPU 显存早已无法容纳完整的模型权重与 KV 缓存。于是分布式推理成了必选项，常见的并行策略包括张量并行（Tensor Parallelism, TP）、流水线并行（Pipeline Parallelism, PP）与专家并行（Expert Parallelism, EP）。TP 将单层的权重矩阵切分到多个 GPU，每次 GEMM 后需要触发 AllReduce 聚合部分结果；EP 则把 MoE 中不同的专家归属到不同 GPU，根据 token 的门控选择进行跨卡 token 重分布（All‑to‑All 通信）。

本文聚焦 TensorRT‑LLM 在**专家并行**上的通信原语与优化，深入其基于 NVLink 的单边 All‑to‑All 实现、流重叠技巧，以及与计算融合的方案。所有结论均源自对代码仓库的静态分析，每个优化点都附有具体文件路径与行号。

## 2. 核心设计与实现：基于 NVLink 的单边 All‑to‑All

在 MoE 层中，依次执行 **dispatch** 和 **combine** 两次 All‑to‑All —— dispatch 将输入的 token 按 router 选择的 top‑k 专家分发到对应 EP rank，combine 则将各专家输出归并回原路由 rank。传统 All‑to‑All 依赖 NCCL 的 `send` / `recv`，需要 CPU 参与控制流，并且在 dispatch 阶段必须显式交换 token 数量才能规划偏移，延迟较高。

TensorRT‑LLM 在 Python API 层（`tensorrt_llm/_torch/distributed/moe_alltoall.py`）实现了一套绕过 CPU 的单边方案。核心数据结构如下：

```python
# tensorrt_llm/_torch/distributed/moe_alltoall.py:177-184
mnnvl_mem = MnnvlMemory(mapping, workspace_size_per_rank)
workspace = mnnvl_mem.as_torch_strided_tensor(torch.uint8)
metainfo = torch.ops.trtllm.moe_a2a_initialize(
    workspace, self.ep_rank, self.ep_size, self.max_num_tokens,
    self.eplb_stats_num_experts)
```

`MnnvlMemory` 在所有 EP rank 间分配出一块**可通过 NVLink 远程直接访问（RDMA）的连续显存**。`workspace` 是一个 `torch.uint8` 张量，它映射到该共享区域，使得每个 rank 都可以通过普通的 GPU load/store 指令读写其他 rank 的 workspace，无需任何 CPU 侧同步。`metainfo` 是一个预先填充好的元数据数组，包含每个 rank 的 send/recv 计数器、偏移、锁标志等，整个 dispatch 过程只依赖它完成地址计算，实现**无锁、零 CPU 参与的 All‑to‑All**。

dispatch 的本质就是每个 rank 依据本地 token 所选的目标专家号，计算出目标 rank 及其在 workspace 中的写入偏移，然后直接通过 NVLink 写入目标 rank 的 workspace 对应位置。这个过程在一次 C++/CUDA op 调用中完成：

```python
# tensorrt_llm/_torch/distributed/moe_alltoall.py:260-278
recv_tensors, combine_payload_offset, eplb_gathered_stats = torch.ops.trtllm.moe_a2a_dispatch(
    token_selected_experts,
    input_payloads,
    self.workspace,
    self.metainfo,
    runtime_max_tokens_per_rank,
    self.ep_rank,
    self.ep_size,
    self.top_k,
    self.num_experts,
    eplb_local_stats,
)
```

`moe_a2a_dispatch` 内部是一个 GPU kernel，它读取本地的 `token_selected_experts`（shape: `[num_tokens, top_k]`），对每个 token 计算出目标 rank，累加该 rank 的已发送 token 数（通过 `metainfo` 中的原子计数器），得到 payload 应该写入的偏移地址。然后它把 `input_payloads`（即该 token 的隐藏层向量）直接 store 到目标 rank 的 workspace 中（地址 = 目标 rank 的 workspace 基址 + 偏移）。combine 阶段类似，只是方向相反：每个 rank 从自己的 workspace 中读取其他 rank 存放的输出 payload，根据 `combine_payload_offset` 写回本地输出张量。

从内存布局看，workspace 是一个形状为 `[ep_size, max_num_tokens, hidden_size]` 的大数组（`uint8`），每个 rank 的“收件箱”是一个连续的 slice。dispatch 和 combine 完全通过这个共享区域交换数据，所有地址计算都在 GPU kernel 里完成，**没有隐式的 memcpy、没有内核启动之间的 CPU 等待，只有在 NVLink 上的读/写请求**。这比使用 NCCL send/recv 需要先通信 counts 再通信数据的两步过程减少了数倍的延迟，并且避免了 NCCL 内部可能的额外拷贝。

`metainfo` 除了保持计数器外，还会携带有用负载的偏移信息，使 combine 时无需再次读取 payload 头部。`dispatch` 返回的 `combine_payload_offset` 是一个整数，指示本次 dispatch 中本地写入 workspace 的位置，combine 便可以直接从这个偏移开始读取。从而实现 dispatch‑combine 的语义配对。

## 3. 优化技术细节

### 3.1 避免写入‑写入冲突的异步发送

TP 或 PP 的 send/recv 常需要借助独立的 CUDA stream 实现异步通信。但 TensorRT‑LLM 的 `communicator.py` 发现：若发送的张量来自用户缓冲池（如 userbuffers），其底层内存在 NCCL send 真正完成前可能被后续操作释放或覆盖，酿成 “write‑write hazard”。解决方案是在发送前 `clone` 一份副本，将通信负载与原始内存解耦：

```python
# tensorrt_llm/_torch/distributed/communicator.py:806-816
if torch.cuda.is_current_stream_capturing():
    self.nccl_comm.send(tensor, dest)
    return

# If the tensor is allocated from non-default memory pool
# like userbuffers, its underlying memory may be reused
# before the send operation is completed.
# We clone the tensor to avoid write-write conflicts.
tensor = tensor.clone()
self.send_stream.wait_stream(torch.cuda.current_stream())
with torch.cuda.stream(self.send_stream):
    self.nccl_comm.send(tensor, dest)
```

这段代码解释了为何 clone 必不可少：在 CUDA graph 捕获期间，图本身已经建立了依赖关系，无需 clone；但正常执行时，`clone` 显式分配一块新显存，发送流拿到独立副本后，主线程便可安全地重用原始 tensor，即使 buffer pool 回收该内存也不会造成 NCCL 写出脏数据。代价是一次额外的显存带宽消耗。

### 3.2 Side Stream 实现专家计算与通信重叠

MoE 层中存在“共享专家”（shared expert）和普通专家两类计算。若所有计算都在主 CUDA stream 上串行执行，NVLink 的 All‑to‑All 通信时段会成为空闲 bubble。TensorRT‑LLM 的 MoE 插件通过 TensorRT 的 SideStream 机制，将部分专家计算迁移到独立的 CUDA stream 上，并与主 stream 上的通信重叠：

```cpp
// cpp/tensorrt_llm/plugins/mixtureOfExperts/mixtureOfExpertsPlugin.cpp:474-488
if (useSideStream())
{
    // Prepare the side stream
    if (!mSideStreamPtr)
    {
        auto const resource_name = nvinfer1::pluginInternal::SideStream::getResourceKey(mSideStreamId);
        nvinfer1::pluginInternal::SideStream side_stream{};
        mSideStreamPtr = reinterpret_cast<nvinfer1::pluginInternal::SideStream*>(
            getPluginRegistry()->acquirePluginResource(resource_name.c_str(), &side_stream));
    }
    // ...stallMainStream...
    mSideStreamPtr->waitMainStreamOnSideStream(stream);
    // Provide data dependency for the shared experts running after this plugin by copying inputs on the main stream
    size_t count = 1;
    for (int i = 0; i < inputDesc[getInputDummyTensorIndex()].dims.nbDims; ++i)
    {
        count *= inputDesc[getInputDummyTensorIndex()].dims.d[i];
    }
    // ...
}
```

核心原理是：主 stream 执行完部分专家（通常是非共享专家）后，将结果通过 All‑to‑All dispatch 写入 workspace，同时通过一个 dummy tensor 的拷贝在 side stream 上建立数据依赖。Side stream 上的共享专家计算所需的输入是从主 stream 拷贝出来的，可以认为它“看到”了主 stream 的最新结果。当主 stream 进行 combine 通信时，side stream 也在并发执行共享专家的前向计算，从而隐藏一部分计算延迟。代价是增加了 stream 同步的复杂度，且需要 GPU 具备充足的并行执行能力，如果 kernel 过小则可能反而因调度开销降低效率。

### 3.3 多流异步下的张量并行 AllReduce

虽然本文重点在 EP，但 TP 的 AllReduce 同样经历了通信与计算的重叠优化。在 TensorRT‑LLM 的代码中（文件未直接提供，但根据 architecture 可推断），通过将 AllReduce 拆分为 reduce‑scatter + all‑gather，并将 reduce‑scatter 嵌入 GEMM 的输出阶段，使得 GEMM 完成一个 tile 即可启动 reduce 通信，同时下一个 tile 仍在计算，形成细粒度的重叠。此外，利用 userbuffers（一组预分配的 shared memory buffer 作为 NCCL 的发送/接收缓冲区）可以减少内部拷贝。这些手法与上述 clone 保护、side stream 等方法共同构成了 TensorRT‑LLM 的分布式通信优化矩阵。

## 4. 性能与权衡

基于 NVLink 的单边 All‑to‑All 设计最大限度地榨取了硬件能力：

- **零 CPU 开销**：dispatch/combine 完全由 GPU kernel 执行，无需每次通信都走 host‑to‑device 同步。在解码阶段单 token 路由时，一次 `.item()` 调用就可能引发数十微秒的延迟，该设计完全规避了这个问题。
- **内存开销的权衡**：workspace 必须按 `ep_size × max_num_tokens × hidden_size` 预先分配，对于大 EP 尺寸或长序列，这是一笔可观显存占用。例如 EP=8、max_tokens=4096、hidden=4096（bf16）需约 512 MB。在 batch 较小时存在内存浪费。
- **与 NCCL 的比较**：NCCL 的 All‑to‑All 内部会进行多步协商（主要是交换 counts 和 displacements），而本方案利用 `metainfo` 将这些信息预置，省去了协商步骤。但这种方式要求 token 总数不超过 `max_num_tokens`，如果超出则必须回退到动态通信模式，增加了工程复杂度。
- **Side stream 的适用边界**：重叠计算与通信仅在专家计算耗时足够长时才有收益，若模型较小或 batch 极小，stream 同步的开销可能反超节省的时间。实际部署时需根据 profiling 结果决定是否启用。

整体上，TensorRT‑LLM 的这一套设计对 DeepSeek‑V2/V3 等大规模 MoE 模型极为友好，能充分利用 NVSwitch 和 NVLink 的高带宽、低延迟特性，在单节点 8 GPU 的推理中获得接近线性的吞吐扩展。

## 6. 总结

TensorRT‑LLM 的专家并行通信以 NVLink 单边 All‑to‑All 为核心，将传统的多步协商压缩为单次 GPU kernel 调用，并通过 workspace 共享、metainfo 预置、side stream 重叠等手段，将 EP 的通信开销降至接近硬件极限。对于想要在 8 GPU 节点上流畅运行百亿参数 MoE 的工程师来说，这些实现细节提供了可直接借鉴的参考——从内存布局到流管理，每一层优化都有明确的代码落点。随着 NVLink 带宽的持续提升和 H100 等新硬件的 FP8 加速，类似的设计理念也将继续演进，支撑起下一波更大规模的分布式推理。
