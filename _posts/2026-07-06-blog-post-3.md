## 1. 背景与目标

大语言模型的核心运算是自注意力，其计算量和显存需求与序列长度的平方成正比。对于长上下文，传统注意力实现会将中间矩阵 \( S = QK^T \) 完整写回 HBM，导致带宽瓶颈和显存溢出。推理场景中，KV cache 的大量碎片化内存请求进一步恶化了这一问题。为了解决这些瓶颈，工业界出现了三类主流优化思路：**FlashAttention** 通过 IO‑aware 的 tile 计算避免写出完整注意力矩阵；**PagedAttention** 将 KV cache 切分成固定块，以虚拟内存的方式管理，消除碎片；**稀疏注意力** 则主动丢弃低分 token，仅计算高相关度的注意力。TensorRT‑LLM 将这三者融合在一个统一的注意力模块中，并在 kernel 层面进行了深度的内存布局、页表机制和精度编排优化。

本文基于 TensorRT‑LLM 开源代码，揭示 FlashAttention、PagedAttention 和稀疏注意力在 NVIDIA GPU 上的具体实现原理，聚焦于 tiling 调度、块偏移索引、indexer 的 top‑k 选择以及可配置的精度控制，最终比较它们的性能权衡。

## 2. 核心设计与实现

### 2.1 FlashAttention：分块计算与 Warp Specialization

FlashAttention 的核心思想是将 \( Q \) 切分成小块（tiles）存储在共享内存或寄存器中，迭代地加载 \( K,V \) 的分块，并在片上完成 softmax 的增量计算，从而避免将 \( QK^T \) 矩阵完整写回 HBM。在 TensorRT‑LLM 中，该机制通过 `determine_launch_params` 函数动态启用，并针对不同 GPU 架构选择最优 tiling 策略。

```cpp
// cpp/kernels/fmha_v2/src/fused_multihead_attention.cpp:512-516
launch_params.flash_attention = (data_type == DATA_TYPE_FP16 ||
    data_type == DATA_TYPE_BF16 || data_type == DATA_TYPE_E4M3)
    && (s >= 16 && d >= 16) && !force_non_flash_attention;
launch_params.warp_specialization = (data_type == ...)
    && sm == 90 && launch_params.flash_attention
    && !force_non_warp_specialization;
```

这段代码在每次启动 attention kernel 前根据序列长度 s、头维度 d、数据类型以及 GPU 的 SM 版本决定是否采用 FlashAttention，并进一步决定是否开启 Warp Specialization。只有当 s≥16 且 d≥16 时才启用，因为过小的矩阵会由于线程同步开销而抵消 IO 节省。

**Warp Specialization** 是 Hopper 架构（SM 90）的特性，利用硬件线程块（Thread Block）内部的 warp 分组实现计算与访存的重叠。具体地，一个 warp 专门负责通过 TMA（Tensor Memory Accelerator）异步拷贝下一个 K,V tile，另一个 warp 负责当前 tile 的 QK^T 计算和 softmax 更新，而其他 warp 则并行处理 V 的乘累加。这种流水线编排通过 `fused_multihead_attention_v2` kernel 中的共享内存队列实现，使得全局内存访问几乎完全隐藏在计算之后。代价是增加了 kernel 的复杂度和 warp 调度开销，尤其对于短序列可能得不偿失。

FlashAttention 的 tile 大小是性能关键。在 Ampere 上，通常将 Q 的 tile 大小设为 `[B, M, H, D]` 中的 M=128 或 64，K tile 大小对应 N，具体值由 kernel 模板在编译期根据 head_dim 和 SM 共享内存容量决定。这些 tile 持留在寄存器中，通过循环展开和向量化加载指令（LDG.128）来最大化带宽利用率。

### 2.2 PagedAttention：页表映射与 KV 块池

PagedAttention 将每个序列的 KV cache 划分为固定大小的块（block），每个块在物理上连续存放，逻辑上通过块偏移表（block table）映射。这让不同的序列可以共享物理块（例如前缀缓存），且避免了长序列导致的显存碎片。TensorRT‑LLM 的实现抽象为 `Kv_block_array` 结构体，并在 kernel 参数中用 `paged_kv_cache` 传递。

```cpp
// cpp/kernels/fmha_v2/src/fused_multihead_attention.cpp:250-270
if (input_layout == Attention_input_layout::Q_PAGED_KV) {
    int max_blocks_per_sequence = (s_kv + tokens_per_block - 1) / tokens_per_block;
    params.paged_kv_cache = Kv_block_array(b, max_blocks_per_sequence,
        tokens_per_block,
        get_size_in_bytes(tokens_per_block * h_kv * std::gcd(d, dv), data_type),
        paged_kv_pool_ptr);
    params.paged_kv_cache.mBlockOffsets = paged_block_offsets;
    params.k_stride_in_bytes = get_size_in_bytes(tokens_per_block * d, data_type);
    params.v_stride_in_bytes = get_size_in_bytes(tokens_per_block * dv, data_type);
}
```

这里 `tokens_per_block` 默认取 64 或 128，物理池 `paged_kv_pool_ptr` 是一段预先分配的设备内存，`mBlockOffsets` 是一个 int32 数组，长度等于 `batch × max_blocks_per_sequence`。在 kernel 内部，对于某个 token `i`，其逻辑块索引为 `block_id = i / tokens_per_block`，物理块基址为 `block_base = mBlockOffsets[block_id]`，token 在块内的偏移为 `offset = i % tokens_per_block`。最终 K、V 地址计算如下：

```
K_ptr = paged_kv_pool_ptr + block_base * k_stride + offset * (head_dim * dtype_size) * h_kv
V_ptr = paged_kv_pool_ptr + block_base * v_stride + offset * (head_dim_v * dtype_size) * h_kv
```

对于 GQA（分组查询注意力），`h_q_per_kv = h / h_kv`，在访问 K,V 时每个 KV 头会被多个 Q 头复用，核内会根据 `h_q_per_kv` 对 Q 进行分组，循环读取同一份 K,V tile，从而减少全局内存读取量。

PagedAttention 的代价是每次访问 KV 都需要一次额外的块表查找，且物理上不连续的块访问可能降低 L2 缓存命中率。为了缓解这一问题，TensorRT‑LLM 采用了较大的块大小（如 64 tokens），使得连续访问的 token 大多落在同一物理块内；同时为每个块表查找使用了最近使用的 block_id 缓存寄存器，尽量减少重复加载。

### 2.3 稀疏注意力：DSA 的 Indexer 与 Chunked Prefill

动态稀疏注意力（DSA）通过对每个查询 token 动态选择最多 top‑k 个键 token 来计算注意力，从而将计算复杂度降为 O(LK)。TensorRT‑LLM 的 DSA 实现包含一个轻量级的 indexer，用 MHA 或 MQA 的全维度 logits 估计每个 Q token 的注意力分布，然后只对 top‑k 个位置进行精确注意力计算。该过程分为两个阶段：indexer 产生 top‑k 索引，然后 sparse attention kernel 根据索引读取 KV 并计算输出。

为了控制 prefill 阶段的显存峰值（indexer 的 logits 矩阵大小为 L×H），DSA 采用了 **Chunked Prefill** 调度：

```python
# tensorrt_llm/_torch/attention_backend/sparse/dsa.py:340-360
num_q_blocks = (seq_len + max_chunk_size - 1) // max_chunk_size
for q_block_idx in range(num_q_blocks):
    token_start = q_block_idx * max_chunk_size
    token_end = min(token_start + max_chunk_size, seq_len)
    # 后续为该 Q 块计算 indexer logits 并提取 topk
```

这段 Python 代码在 host 端将长序列的 Q 拆分成多个 Q‑blocks（每个最多 `max_chunk_size` 个 token），每个 Q‑block 独立运行 indexer，使得中间 logits 矩阵尺寸缩小到 `[chunk_size, num_heads]`，大幅降低显存占用。对于多个短序列，则尝试将其打包进同一个 chunk，共享一次 indexer 调用，减少 kernel 启动开销。

Indexer 的输出是 top‑k 索引（int32 张量），作为稀疏注意力 kernel 的输入。解码阶段还可以利用 **Heuristic Top‑K** 加速：将上一步的 top‑k 索引偏移一个时间步作为当前的预排序 hint，kernel 沿着 hint 顺序快速筛选，将排序复杂度从 O(L log L) 降至接近 O(K)。该 hint 跨步依赖，需要每层维护一个 `prev_topk` 状态。

```python
# tensorrt_llm/_torch/attention_backend/sparse/dsa.py:1725-1730
if self._enable_heuristic_topk:
    local_layer = metadata.kv_cache_manager.layer_offsets[self.layer_idx]
    pre_idx = metadata.heuristic_prev_topk[local_layer, :num_generations]
    heuristic_scratch = metadata.heuristic_scratch_values[:num_gen_tokens]
```

这里 `pre_idx` 传入 CUDA kernel 作为初始搜索路径，若 hint 失效（例如上下文突变）则退化为常规全排序。

## 3. 优化技术细节

### 3.1 Skip Softmax 阈值

在 PagedAttention 和稀疏注意力 kernel 中，可以设置跳过热力阈值：当 `QK^T` 的值乘以 `scale_bmm1` 后小于某个阈值时，这些 logits 对 softmax 的贡献可忽略，可直接当作 0 处理，跳过 exp 计算和后续的 V 乘加。这显著减少了对于 padding 或极低分值的无效计算。

```cpp
// cpp/tensorrt_llm/kernels/decoderMaskedMultiheadAttention/decoderXQARunner.cpp:513-517
if (isSkipSoftmax) {
    TLLM_CHECK_WITH_INFO(isGMMAKernel,
        "skip softmax is only supported for GMMA kernel for now.");
    ...
    appendParam(&xqaParams.skip_softmax_threshold_scale_factor);
}
```

该参数是一个浮点标量，kernel 内部会比较 `QK * scale - threshold` 是否小于 0，若小于则将该位置的 exp 设为 0。阈值通常通过校准或根据模型分布设定，牺牲少量精度换取吞吐提升。

### 3.2 FP8/FP4 量化下的扁平 slot mapping

为了支持 FP8 和 FP4 精度的 KV cache，TensorRT‑LLM 将数据 byte 和 scale 连续存放，并为每个 token 预计算 **扁平字节索引**，从而实现向量化的 scatter/gather 操作，避免复杂的坐标转换。

```python
# tensorrt_llm/_torch/attention_backend/sparse/dsa.py:295-305
fp8_indices = block_ids * block_stride + pos_in_blocks * data_bytes_per_token
scale_indices = block_ids * block_stride + scale_base_offset + pos_in_blocks * scale_size
```

`block_ids` 是一个 int32 张量，保存每个 token 所在物理块的偏移，`block_stride` 是每个块的总字节数（包含 K 数据和 scale 区域）。通过简单的乘法和加法就得到 flat index，scatter kernel 可以直接使用 `cudaMemcpyAsync` 或自定义的 128‑位向量化写入，将量化后的 KV 数据写入池中对应位置。这种扁平布局提高了内存吞吐和 cache 操作的可编排性，但要求 KV cache 的存储格式完全固定。

### 3.3 GQA/MQA 的步幅调整

在多查询（MQA）和分组查询注意力中，K、V 的头数量少于 Q 头数量，通过设置 `h_q_per_kv` 和调整 stride 实现共享。在 `set_params` 中，当输入布局为 `PACKED_QKV` 时，K、V 的 stride 被计算为仅包含自己的 KV 头数对应的字节数，而非完整的 QKV 大小。

```cpp
// cpp/kernels/fmha_v2/src/fused_multihead_attention.cpp:240-250
params.h_kv = h_kv;
params.h_q_per_kv = h / h_kv;
if (input_layout == Attention_input_layout::PACKED_QKV) {
    params.k_stride_in_bytes = get_size_in_bytes(
        h * d + h_kv * d + h_kv * dv, data_type);
}
```

在 kernel 内，每个线程块处理一个 Q 头组，K、V 的 tile 只需加载一次，被组内所有 Q 头复用，从而将 KV cache 的读取量降至原来的 `h_kv / h` 倍。这种 layout 调整是 GQA 高效执行的基础。

## 4. 性能与权衡

| 技术 | 性能增益 | 主要代价 | 适用场景 |
|------|---------|---------|---------|
| FlashAttention | 避免 O(L²) 中间矩阵显存占用，减少 HBM 访问，提升长序列 prefill 吞吐 | 增加了 tiling 调度和同步开销，对极小序列可能无收益；Warp Specialization 仅 Hopper 可用 | 长上下文预填充 |
| PagedAttention | 消除 KV cache 碎片，支持动态内存分配和块级共享，大幅提高 serving 吞吐 | 块表查找增加延迟，不连续访问降低 L2 命中率；块大小需权衡利用率与表大小 | 高并发在线推理 |
| 稀疏注意力 | 计算量降为 O(LK)，允许更大 batch 和更长的上下文 | Indexer 开销，top‑k 可能遗漏关键 token，精度受 K 与阈值影响；状态维护复杂 | 长上下文解码 |
| Skip Softmax | 减少无价值计算 | 阈值选择敏感，可能损失精度 | 存在大量低分区域时 |
| Heuristic Top‑K | 解码排序加速 | 分布突变时回退，状态维护 | 自回归解码 |

所有这些优化在 TensorRT‑LLM 中可通过配置和运行时参数按需启用或关闭，实现了“精度可编排”——用户可以在吞吐和精度之间灵活调节。

## 6. 总结

TensorRT‑LLM 中的注意力模块将 FlashAttention 的分块计算、PagedAttention 的页表映射和稀疏注意力的动态 top‑k 紧密耦合，形成了一套高度可配置的注意力计算基础设施。从 tile 调度到 warp 级流水线，从块偏移表到扁平向量化访存，每一个细节都反映了 GPU 上内存层级和计算模型的极致运用。工程师可以在不同的业务场景下灵活启用这些技术，并对精度进行编排，在吞吐和延迟间寻找最优平衡。理解这些底层实现原理，有助于在自研推理框架时做出更合理的设计选择。
