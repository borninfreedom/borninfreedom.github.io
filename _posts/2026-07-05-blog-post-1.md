## 一、全局架构概览

openclaw 的通信系统是一个"中央集权 + 外部插件"的混合架构。一张图概括:

```
┌────────────────────────────────────────────────────────────────────┐
│                         本机 (Mac/Linux)                          │
│                                                                    │
│   ┌──────────────────────────────────────────────────────────┐    │
│   │                  Gateway 进程 (默认 18789)               │    │
│   │                                                          │    │
│   │  ┌────────────┐  ┌──────────────┐  ┌────────────────┐   │    │
│   │  │ HTTP Server │  │ WebSocket    │  │ Channel Plugin │   │    │
│   │  │ /health     │  │ Server (ws)  │  │ Runtime        │   │    │
│   │  │ /v1/chat    │  │ req/res/event│  │ (微信/Discord) │   │    │
│   │  │ /tools/invoke│ │ 自定义三帧   │  │ in-process     │   │    │
│   │  └────────────┘  └──────────────┘  └────────────────┘   │    │
│   │                                                          │    │
│   │  ┌────────────────────────────────────────────────┐      │    │
│   │  │  Agent Runtime (in-process)                    │      │    │
│   │  │  └─→ LLM Provider (DeepSeek/OpenAI/Anthropic)  │      │    │
│   │  └────────────────────────────────────────────────┘      │    │
│   └──────────────────────────────────────────────────────────┘    │
│                          ↑↓ (loopback)                            │
│   ┌────────────────┐  ┌──────────────┐  ┌──────────────────┐     │
│   │ CLI / TUI      │  │ Control UI  │  │ launchd / systemd │     │
│   │ (WS client)    │  │ (浏览器)    │  │ (启动 gateway)    │     │
│   └────────────────┘  └──────────────┘  └──────────────────┘     │
└────────────────────────────────┬───────────────────────────────────┘
                                 │ 出站 HTTPS (long-poll)
                                 ↓
                  ┌──────────────────────────────┐
                  │  ilinkai.weixin.qq.com       │
                  │  (Tencent iLink Bot API)      │
                  └──────────────────────────────┘
                                 ↑↓ HTTPS
                  ┌──────────────────────────────┐
                  │  novac2c.cdn.weixin.qq.com   │
                  │  (媒体 CDN,AES-128-ECB 加密)│
                  └──────────────────────────────┘
                                 ↑
                  ┌──────────────────────────────┐
                  │   微信客户端(你的手机)       │
                  └──────────────────────────────┘
```

关键观察:

1. **gateway 是唯一监听端口的进程**,默认绑 `127.0.0.1:18789`,同时承载 HTTP 和 WebSocket
2. **微信通道是 gateway 进程内的子模块**,不是独立进程
3. **整个微信通信路径都是出站 HTTPS**(long-poll + CDN),不需要公网 IP,不需要 webhook
4. **agent 也在 gateway 进程内**,通过 in-process 调用,不 spawn 子进程

这个设计的核心哲学:**让 openclaw 主动连接外部世界,而不是让外部世界能找到 openclaw**。这与传统的 webhook-based bot 架构完全相反。

## 二、Gateway:中央枢纽

### 2.1 Gateway 是什么

Gateway 是 openclaw 的**单一中央服务进程**,既是 HTTP 服务器,也是 WebSocket 服务器,承担"操作平面 / 控制平面 / 通道编排 / agent 调度"的中枢角色。

入口源码:

- 公共启动器:``openclaw/src/gateway/server.ts`` — `startGatewayServer()` 懒加载实现
- 真正实现:``openclaw/src/gateway/server.impl.ts`` — `startGatewayServer` 在第 24 行导出
- CLI 命令入口:``openclaw/src/cli/gateway-cli/run.ts:818``

核心职责:

1. 维护 WS 客户端连接池,做握手 / 认证 / 限流
2. 注册并 dispatch 一组 RPC 方法
3. 加载 channel 插件,把 inbound 消息转成 agent 调用,outbound 消息发回各通道
4. 提供 HTTP 端点:健康探针、OpenAI 兼容 API(`/v1/chat/completions`、`/v1/responses`、`/v1/embeddings`、`/v1/models`)、`/tools/invoke`、`/sessions/:id/history`、`/sessions/:id/kill`、Control UI、webhook hooks、MCP HTTP 等
5. 调度 cron / heartbeat / plugin services

默认端口 `18789`,定义在 ``openclaw/src/config/paths.ts:262``:

```typescript
export const DEFAULT_GATEWAY_PORT = 18789;
```

端口可由 `OPENCLAW_GATEWAY_PORT` 环境变量或 `gateway.port` 配置覆盖。

### 2.2 默认绑 127.0.0.1 的设计

绑定策略在 ``openclaw/src/gateway/net.ts:262-317`` 的 `resolveGatewayBindHost()`:

| 模式 | 绑定地址 | 适用场景 |
|---|---|---|
| `loopback`(默认) | `127.0.0.1` | 本机使用 |
| `lan` | `0.0.0.0` | 局域网访问 |
| `tailnet` | Tailnet IPv4 | Tailscale 内网 |
| `auto` | 容器内 `0.0.0.0`,裸机 `127.0.0.1` | 自动判断 |
| `custom` | 用户指定 IPv4,失败 fallback 到 `0.0.0.0` ⚠️ | 自定义 |

**为什么默认 loopback**:gateway 默认承载 token / password 等敏感认证,loopback 绑定避免凭据被局域网嗅探。非 loopback 时会显式告警(``server-runtime-state.ts:211-216``)。

### 2.3 HTTP 与 WebSocket 双协议

HTTP 和 WebSocket **共享同一端口**:

- HTTP 服务器在 `createGatewayHttpServer` 中创建(``server-http.ts:473``)
- WebSocket 服务器 `new WebSocketServer({ noServer: true })`,通过 `attachGatewayUpgradeHandler` 挂在 HTTP server 的 `upgrade` 事件上

所以一个端口同时承载 HTTP 和 WS 流量,通过 HTTP `Upgrade` 头区分。

### 2.4 WebSocket 协议详解

实现库:**原生 `ws` 包**(不是 socket.io)。

消息格式是自定义的三帧类型,定义在 ``openclaw/src/gateway/protocol/schema/frames.ts:138-168``:

```typescript
type Frame =
  | { type: "req"; id: string; method: string; params: unknown }     // 请求帧
  | { type: "res"; id: string; ok: boolean; payload?: unknown; error?: unknown }  // 响应帧
  | { type: "event"; event: string; payload: unknown; seq: number; stateVersion: number };  // 服务端推送
```

**握手流程**(关键 6 步):

1. 服务端 accept 后立刻发送 `connect.challenge` 事件,带 `nonce` 和 `ts`(``ws-connection.ts:313-318``)
2. 客户端必须发首帧 `{type:"req", method:"connect", params: ConnectParams}`
3. `ConnectParams` 包含 `minProtocol`/`maxProtocol`/`client`/`role`/`scopes`/`auth`/`device`
4. `auth` 字段可含 `token`/`password`/`bootstrapToken`/`deviceToken`/`approvalRuntimeToken`
5. 服务端做协议版本协商、role 校验、origin 检查、auth 校验
6. 通过后回 `hello-ok` 帧

**注意**:token 通过**握手帧的 `params.auth.token` 字段**传递,而不是 HTTP Header 或 query string。HTTP 端点则用标准 `Authorization: Bearer <token>`。

**心跳 / 重连**:

- 服务端 ping:每 25s `socket.ping()`(``ws-connection.ts:490-496``)
- 客户端:指数退避重连 `backoffMs = min(backoffMs*2, 30_000)`
- tick 监控:`tickIntervalMs * 2` 没收到 tick 则 `close(4000, "tick timeout")`
- 握手超时由 `resolvePreauthHandshakeTimeoutMs` 控制,超时直接关连接

### 2.5 Gateway 的连接拓扑

| 谁连接谁 | 通信方式 | 说明 |
|---|---|---|
| 微信/Discord 等通道 ↔ gateway | in-process 同进程调用 | 通道是 gateway 进程内子模块 |
| agent ↔ gateway | in-process | agent 也在 gateway 进程内,通过 `pi-embedded-runner` 调度 |
| CLI ↔ gateway | WS 连接 `ws://127.0.0.1:18789` | `GatewayClient` 客户端,做 connect 握手 |
| 远程用户 ↔ gateway | 三种方案 | Tailscale / LAN + TLS / 反向代理 |

**远程访问的三种方案**:

1. `gateway.bind="lan"|"custom"` + TLS(``net.ts:287-301``)
2. `gateway.tailscale="serve"|"funnel"` — 推荐,走 Tailscale HTTPS 反代,仍绑 loopback(``server-tailscale.ts``)
3. 反向代理(nginx 等)配合 `gateway.auth.mode="trusted-proxy"`,由代理注入 `X-Forwarded-User` 等头

### 2.6 启动流程(launchd 路径)

macOS 上 gateway 由 launchd 启动:

1. `openclaw gateway install` → `installLaunchAgent`(``launchd.ts:928-945``)
2. 生成 plist: `~/Library/LaunchAgents/ai.openclaw.gateway.plist`,关键 keys:`RunAtLoad=true`、`KeepAlive=true`、`ExitTimeOut=20s`、`ThrottleInterval=10s`、`Umask=077`
3. env 文件: `$STATE_DIR/service-env/ai.openclaw.gateway.env`,权限 `0o600`
4. wrapper 脚本: `ai.openclaw.gateway-env-wrapper.sh`,内容是
   ```sh
   #!/bin/sh
   set -eu
   env_file="$1"; shift
   [ -f "$env_file" ] && . "$env_file"
   exec "$@"
   ```
5. ProgramArguments 包装:`[wrapperPath, envFilePath, ...原始 args]`,即先 source env 文件,再 exec `node dist/index.js gateway --port 18789`

修改 env / `openclaw.json` 后必须重启 gateway:

```bash
launchctl kickstart -k gui/$(id -u)/ai.openclaw.gateway
```

不重启不生效(launchd 进程不感知 env 文件变化)。

## 三、微信通道:为什么 + 怎么做

### 3.1 微信通道是什么

微信通道是 openclaw 的**外部扩展插件**,不是核心内置模块。

- npm 包名:`@tencent-weixin/openclaw-weixin@2.4.3`
- 作者:Tencent
- plugin id 与 channel id 均为 `openclaw-weixin`
- 插件源码:`.openclaw/npm/node_modules/@tencent-weixin/openclaw-weixin/src/`
- 插件 manifest: `…/openclaw.plugin.json`

openclaw 文档 ``openclaw/docs/channels/wechat.md:27-43`` 明确表达设计意图:

> The WeChat code does not live in the OpenClaw core repo. OpenClaw provides the generic channel plugin contract, and the external plugin provides the WeChat-specific runtime.

即 openclaw 核心保持 channel-agnostic,微信相关代码全部交给外部插件。

### 3.2 为什么需要微信通道

`README.md:1-6` 写明用途:

> OpenClaw's WeChat channel plugin, supporting login authorization via QR code scanning.

微信是中国市场主导聊天平台,做这个通道是为了让 AI 员工触达用户日常使用的会话入口。

相比 Discord/Slack/Telegram 的特殊性:

| 维度 | Discord/Slack/Telegram | 微信 |
|---|---|---|
| 官方机器人 API | 有 | **没有**(只能登录个人微信号) |
| 媒体传输 | HTTPS URL 直传 | **AES-128-ECB 加密 CDN** |
| 上下文 token | 不需要 | **必须原样回传 `context_token`** |
| 群聊支持 | 有 | **未开放**(`capabilities.chatTypes: ["direct"]`) |
| 协议 | Webhook 或 WebSocket | **HTTP long-poll** |

`@im.bot` 是 ilink 内部 bot 标识,`@im.wechat` 是 wechat 用户标识,本质上是用 ilink bot 协议登录个人微信。

### 3.3 微信如何接收消息

**主动长轮询(HTTP long-poll),非推送**。这是个人微信号,不是公众号/小程序。

- 协议:HTTPS POST,baseUrl 为 `https://ilinkai.weixin.qq.com`(``accounts.ts:11``)
- 端点:`ilink/bot/getupdates`
- 鉴权:`AuthorizationType: ilink_bot_token` + `Authorization: Bearer <token>`,外加随机 `X-WECHAT-UIN` header(``api.ts:222-253``)
- 长轮询默认 35s(``monitor.ts:13`` `DEFAULT_LONG_POLL_TIMEOUT_MS = 35_000`),服务器按需返回 `longpolling_timeout_ms` 提示下次轮询超时
- 同步游标 `get_updates_buf` 持久化到 `accounts/<id>.sync.json`,重启 gateway 后能从上次位置恢复

监听循环在 ``monitor.ts:42-209``:

```typescript
async function monitorWeixinProvider(...) {
  while (!abortSignal?.aborted) {
    const updates = await getUpdates({ ... });
    for (const msg of updates) {
      await processOneMessage(msg);
    }
  }
}
```

**扫码登录流程**:

1. 终端调 `ilink/bot/get_bot_qrcode?bot_type=3` 拿二维码 URL
2. 用 `qrcode-terminal` 在终端显示二维码
3. 用户用微信扫码
4. 长轮询 `ilink/bot/get_qrcode_status`(35s),经历 `wait` → `scaned` → `confirmed`
5. `confirmed` 时服务端返回 `bot_token`、`ilink_bot_id`(后缀 `@im.bot`)与 `ilink_user_id`(后缀 `@im.wechat`)
6. 账号信息保存到 `.openclaw/openclaw-weixin/accounts/<normalized-id>.json`,内含 `token`、`baseUrl`、`userId`

还支持配对码(`need_verifycode`)与已绑定短路(`binded_redirect`)。

### 3.4 微信如何发送消息

openclaw 调用 Tencent iLink 的 `sendmessage` CGI 把回复推回微信,所有外发均为 HTTPS POST:

- **文本**:`sendMessage` → `apiPostFetch("ilink/bot/sendmessage")`
- **媒体**(图片/视频/文件/语音):
  1. 先 `getUploadUrl`(`ilink/bot/getuploadurl`)拿预签名
  2. 本地 **AES-128-ECB 加密**后 PUT 上 CDN
  3. 再把 `encrypt_query_param` + `aes_key` 包成 `MessageItem` 发送
- **输入状态**:`ilink/bot/sendtyping`,需要先 `ilink/bot/getconfig` 拿 `typing_ticket`

**关键约束**:每条外发必须带回原消息的 `context_token`,token 缓存在内存 + 持久化到 `accounts/<id>.context-tokens.json`。

### 3.5 微信通道与 gateway 的关系

**gateway 进程内的子模块,非独立进程**。

证据:``.openclaw/plugins/installs.json:4643-4648`` 中 `"startup": { "sidecar": false, ... }`,与 `acpx`/`active-memory` 等标记 `sidecar: true` 的插件对比明显。

加载与生命周期:

- gateway 启动时由 `createChannelManager`(``openclaw/src/gateway/server-channels.ts:234``)枚举已启用的 channel plugin
- 对每个 account,gateway 调 `plugin.gateway.startAccount(ctx)`(``channel.ts:395-463``),其中通过 `await import("./monitor/monitor.js")` 动态加载并执行 `monitorWeixinProvider`
- gateway 通过 `ctx.channelRuntime` 注入一个 task-scoped 的 `PluginRuntime["channel"]` 表面,带 reply/routing/session/media/commands 子能力(``server-channels.ts:143-176``)

## 四、内网穿透机制:不需要公网 IP 的设计

### 4.1 核心结论:openclaw 完全不需要公网 IP

整个微信通信路径**全部是 openclaw 主动发起的出站 HTTPS**:

| 通信类型 | 端点 | 方向 |
|---|---|---|
| 长轮询拉消息 | `https://ilinkai.weixin.qq.com/ilink/bot/getupdates` | 出站 |
| 发送回复 | `https://ilinkai.weixin.qq.com/ilink/bot/sendmessage` | 出站 |
| 上传/下载媒体 | `https://novac2c.cdn.weixin.qq.com/c2c` | 出站 |
| 扫码登录 | `https://ilinkai.weixin.qq.com/ilink/bot/get_bot_qrcode` | 出站 |

**不存在反向连接、webhook URL 或端口监听**。openclaw 进程只作 iLink 客户端,适合在 NAT 后方部署。

### 4.2 为什么不需要内网穿透

传统 bot 架构(如 Telegram bot)通常需要 webhook,这意味着:

1. 平台主动 HTTP POST 到你的服务器
2. 你的服务器必须有公网可达的 URL
3. 内网部署需要 frp / ngrok / cloudflare tunnel 等穿透工具

openclaw 的设计完全相反:

- **长轮询代替 webhook**:gateway 进程持续保持一个 35s 的 HTTP 请求到 iLink 服务器,服务器有消息时立即返回,否则超时返回,客户端立即重连
- **"伪长连接"**:靠服务端 hold 请求 35s + 客户端立即重连实现
- **辅以退避策略**:`BACKOFF_DELAY_MS = 30_000` 与 `MAX_CONSECUTIVE_FAILURES = 3` 保证鲁棒性
- **session 失效保护**:errcode `-14` 时进入暂停态避免雪崩

### 4.3 这套机制的网络属性

| 属性 | 表现 |
|---|---|
| NAT 友好 | ✅ 完全友好,只要能出站 HTTPS 即可 |
| 防火墙穿透 | ✅ 标准出站 443,无需特殊规则 |
| 代理支持 | ✅ 走系统代理或环境变量 |
| 公网 IP 需求 | ❌ 完全不需要 |
| Webhook URL 需求 | ❌ 完全不需要 |
| 端口转发需求 | ❌ 完全不需要 |

### 4.4 Gateway 监听本地端口的合理性

回到用户的问题:**为什么 gateway 监听本地端口就可以了?**

答案现在已经清楚:

1. **gateway 监听本地端口只是为了本机 CLI / Control UI / HTTP API 客户端访问**,不是为了让外部世界连进来
2. **微信通道在 gateway 进程内**,直接 in-process 调用,根本不经过网络
3. **gateway 主动出站连接微信服务器**,微信服务器不需要反向连回来
4. **agent 也在 gateway 进程内**,in-process 调用 LLM provider

所以 gateway 的本地端口只是给"本机用户"用的(开发者跑 CLI、Control UI 浏览器、自动化脚本调 `/v1/chat/completions`)。微信通道的网络通信完全不依赖这个端口。

### 4.5 真需要远程访问的方案

如果你确实需要从外网访问 gateway(比如出差时手机调 API),有三种方案:

1. **Tailscale Serve/Funnel**(推荐):
   ```json
   { "gateway": { "tailscale": "serve" } }
   ```
   gateway 仍绑 loopback,Tailscale daemon 反代到外部,自动 HTTPS。

2. **LAN 模式 + TLS**:
   ```json
   {
     "gateway": {
       "bind": "lan",
       "tls": { "enabled": true, "autoGenerate": true }
     }
   }
   ```
   强制 TLSv1.3,自动签证书。

3. **反向代理 + trusted-proxy**:
   ```json
   {
     "gateway": {
       "bind": "loopback",
       "auth": { "mode": "trusted-proxy", "trustedProxy": { "userHeader": "X-Forwarded-User" } }
     }
   }
   ```
   nginx/caddy 反代到本机 gateway,由代理注入认证头。

## 五、端到端消息流转全链路

这一节回答最关键的问题:**从"用户在微信发一条消息"到"用户在微信收到回复",中间到底发生了什么?**

### 5.1 完整时序

```
[用户手机微信]
     │ 1. 发送消息 "你好"
     ↓
[微信服务器]
     │ 2. 缓存在 ilinkai.weixin.qq.com
     ↓ (gateway 长轮询拉到)
[gateway: monitorWeixinProvider]
     │ 3. processOneMessage(WeixinMessage)
     ↓
[gateway: 通道插件构建 MsgContext]
     │ 4. buildChannelInboundEventContext → FinalizedMsgContext
     │    (sender / conversation / route / message / access / command)
     ↓
[通道插件: resolveAgentRoute]
     │ 5. 按 (channel, accountId, peer) 匹配 cfg.bindings
     │    → ResolvedAgentRoute { agentId, sessionKey }
     ↓
[通道插件: runInboundReplyTurn]
     │ 6. 调用 runChannelTurn (kernel.ts:596)
     ↓
[kernel: ingest]
     │ 7. adapter.ingest(raw) → NormalizedTurnInput
     │    (id / rawText / textForAgent)
     ↓
[kernel: classify]
     │ 8. adapter.classify(input) → ChannelEventClass
     │    判断是否要触发 agent turn (reaction/lifecycle 不触发)
     ↓
[kernel: preflight]
     │ 9. adapter.preflight(input, eventClass) → PreflightFacts
     │    - allowlist 检查 (allow-from.ts / allowlist-match.ts)
     │    - mention / command 检查
     │    - 可返回 drop / observeOnly / handled
     ↓
[kernel: resolveTurn]
     │ 10. adapter.resolveTurn(input, eventClass, preflight)
     │     → AssembledChannelTurn
     │     组装 ctxPayload / recordInboundSession / dispatchReply / delivery
     ↓
[kernel: dispatchAssembledChannelTurn]
     │ 11. resolveBotLoopProtectionDrop (bot-loop 防护)
```

### 5.2 关键机制详解

#### 5.2.1 入站去重(durable receive journal)

``durable-receive.ts`` 提供 `createDurableInboundReceiveJournal`,通过 `pendingStore` + `completedStore` 两个 keyed store(带 TTL)实现幂等接收:

- `accept(id, payload)` 返回 `accepted`/`pending`/`completed` 三态
- `complete(id)` 写入 completedStore 并删除 pendingStore
- `release(id)` 增加尝试计数并保留 pending(用于重试)

这让通道能在 webhook 红包重投或进程崩溃后恢复时去重。

#### 5.2.2 入站节流(debounce)

``inbound-debounce-policy.ts`` 与 ``auto-reply/inbound-debounce.ts`` 在按 `buildKey` 分桶的 buffer 中暂存连续消息,`debounceMs` 到期后调用 `onFlush(items)` 一起处理。

控制命令、媒体消息不走 debounce。

#### 5.2.3 ACK 策略

``receive.ts:5`` 定义:

```typescript
type MessageAckStage = "receive_record" | "agent_dispatch" | "durable_send" | "manual";
```

`MessageReceiveContext` 暴露 `ack()`/`nack(err)`,由 `shouldAckAfter(stage)` 决定何时向平台 ack(避免过早 ack 丢消息)。

#### 5.2.4 路由解析

``resolve-route.ts`` 的 `resolveAgentRoute` 遍历 `cfg.bindings`,按 `(channel, accountId, peer.kind, peer.id, guildId, teamId, memberRoleIds)` 匹配。命中优先级:

```
binding.peer > binding.peer.parent > binding.peer.wildcard
            > binding.guild+roles > binding.guild > binding.team
            > binding.account > binding.channel > default
```

会话 key 生成:`buildAgentSessionKey` 按 `dmScope`(`main`/`per-peer`/`per-channel-peer`/`per-account-channel-peer`)拼出 `<agentId>:<channel>:<accountId>:<peerKind>:<peerId>` 形式的稳定 key,保证同一用户跨消息落到同一 session。

#### 5.2.5 出站 draft preview / streaming

``live.ts`` 的 `LiveMessageState` 跟踪 `phase: "idle" | "previewing" | "finalized" | "cancelled"`。

`deliverFinalizableLivePreview` 让通道在最终 payload 到达时直接 `editFinal(previewId, edit)` 原地编辑预览消息(Slack/WhatsApp/Telegram 编辑消息)。编辑失败时按 `handlePreviewEditError` 决定 `fallback`(重发新消息)或 `retain`(保留预览)。

#### 5.2.6 错误处理

**bot loop 保护**(``bot-loop-protection.ts``):

- 以 `(scopeId, conversationId, senderId, receiverId)` 为键,滑动窗口记录事件数
- 默认 `maxEventsPerWindow=20`、`windowMs=60s`、`cooldownMs=60s`
- 超限后 `recordChannelBotPairLoopAndCheckSuppression` 返回 `suppressed: true`,kernel 直接 `drop: bot-loop-protection`

**durable 投递失败**(``durable-delivery.ts:96-113``):

- `status === "failed"` 时重抛
- 若 `sentBeforeError === true`,通过 `markDurableInboundReplyDeliveryErrorVisible` 把错误打上 `visibleReplySent: true` 标记,避免重复消息

**send 失败分级**(``send.ts:47-102``):

| 状态 | 含义 |
|---|---|
| `sent` | 全部成功 |
| `suppressed` | hook 取消/无可见 payload |
| `partial_failed` | 部分成功后失败,`sentBeforeError: true` |
| `failed` | 全部失败,`stage: "platform_send" / "queue" / "unknown"` |

**agent 运行失败**:`buildKnownAgentRunFailureReplyPayload` 在 agent 调用失败时构造错误回复 payload(上下文溢出、模型不可达),并通过 `resolveFallbackTransition` + `resolveConfiguredFallbackModel` 切换到 fallback 模型重试。`TRANSIENT_HTTP_RETRY_DELAY_MS = 2_500` 处理瞬时网络错误。

## 六、网络安全:风险与防御

### 6.1 认证机制全貌

#### 6.1.1 Token 生成、存储、校验

**Token 解析**(``auth-token-resolution.ts:12-85``)优先级:

```
explicit > config (明文) > secretRef > env (OPENCLAW_GATEWAY_TOKEN)
```

当前项目走 SecretRef → `OPENCLAW_GATEWAY_TOKEN`。

**Token 校验**(``auth.ts:354-376``)`authorizeTokenAuth` 调用 ``secret-equal.ts:12-31`` `safeEqualSecret`,内部用 `timingSafeEqual` + 等长补零防时序侧信道。

**速率限制**(``auth-rate-limit.ts:99-235``)默认 10 次/分钟、5 分钟锁定。

**⚠️ 关键风险**:`exemptLoopback` 默认 `true`,意味着 **loopback 来源不限速**,本地攻击者可无限暴力破解 token。

**Token 不会过期、不轮换**:源码中没有 token 过期时间字段,一旦泄露需手动更换。

#### 6.1.2 Token 传输方式

- **HTTP Bearer**: `Authorization: Bearer <token>` (``http-auth-utils.ts:29-35``)
- **WebSocket 握手帧**: `params.auth.token`(不是 HTTP header,不是 query string)
- **Hook 端点禁止 query token**(``hooks-request-handler.ts:187-194``)显式拒绝 `?token=`
- **⚠️ assistant-media 端点允许 query token**(``control-ui.ts:245-274``):会出现在 access log、browser history、Referer header 中,5 分钟 TTL 票据机制部分缓解

#### 6.1.3 日志脱敏

``redact-sensitive-url.ts:46-77`` `redactSensitiveUrl` 会脱敏 URL 的 `token/key/api_key/secret/access_token` 等 14 种敏感参数名。但 **HTTP header 中的 Bearer token 不在此机制覆盖范围**。

### 6.2 传输层安全

#### 6.2.1 HTTP/HTTPS

``infra/tls/gateway.ts:70-72``**默认 HTTP**。只有 `cfg.gateway.tls.enabled === true` 才启用 TLS,且强制 `TLSv1.3`,可自动生成自签证书。

**当前项目 openclaw.json 未配置 tls**,即 token 在本机以明文 HTTP 传输。loopback 模式下这通常可接受,但若误配反代或 LAN 暴露则成问题。

#### 6.2.2 网络绑定

``net.ts:262-317``:

- `loopback`(默认) → `127.0.0.1`
- `lan` → `0.0.0.0`
- `custom` 失败时 **fallback 到 `0.0.0.0`** ⚠️ — 隐蔽风险

``server-runtime-config.ts:140-144`` 强制非 loopback 必须配 token/password,否则启动报错。

#### 6.2.3 CORS / Origin

``origin-check.ts:35-78``:

- `allowlist.has("*")` → 全放行 ⚠️ 危险
- `allowHostHeaderOriginFallback` → Host header 兜底 — DNS rebinding 风险

### 6.3 exec-approvals 安全机制

#### 6.3.1 用途

``exec-approvals.ts:88-100`` 控制哪些 shell 命令 agent 可执行,分三种 `security` 模式:

| 模式 | 行为 |
|---|---|
| `deny` | 全部拒绝 |
| `allowlist` | 必须匹配 `agents.<id>.allowlist` 中的 pattern |
| `full`(默认) | **无限制** ⚠️ |

`allowlist` 模式下命令必须匹配 pattern,或被 socket 协议交互式批准,token 由 `crypto.randomBytes(24).toString("base64url")` 生成。

#### 6.3.2 Symlink 防御

``exec-approvals.ts:266-291`` `ensureDir` 调用 `assertNoExecApprovalsSymlinkParents`,信任根是 `resolveRequiredHomeDir()`(即 `OPENCLAW_HOME` 或 `HOME`)。

**这就是项目规则 ``openclaw_use.md § 1.3`` 软链穿透报错的根因** —— 设计上是对的,防止 attacker 用 symlink 把 `~/.openclaw/exec-approvals.json` 重定向到任意位置。

写文件用 `O_NOFOLLOW | O_EXCL` 防 symlink/hardlink 替换;`assertSafeExecApprovalsOverwriteFallback` 检查 `nlink > 1` 防 hardlink 攻击。文件权限 `0o600`,目录 `0o700`。

#### 6.3.3 攻陷影响

如果 attacker 能写 `exec-approvals.json`:

- 改 `defaults.security` 为 `"full"`(默认就是 full)→ agent 可执行任意命令
- 改 `defaults.ask` 为 `"off"`(默认就是 off)→ 不再询问
- 直接给 `agents.<id>.allowlist` 加 `pattern: "*"` → 所有命令通过

结合 token 盗用,这是完整的 RCE 链路。

### 6.4 Secret 管理

#### 6.4.1 SecretRef

SecretRef 形如 `{source:"env", provider:"default", id:"OPENCLAW_GATEWAY_TOKEN"}`,由 `resolveConfiguredSecretInputString` 解析,值从 env 读取,**配置文件中不含明文**。

#### 6.4.2 明文检测

``audit.ts:42-77`` `SecretsAuditReport` 扫描 4 类问题:

- `PLAINTEXT_FOUND` — 配置中有明文 secret
- `REF_UNRESOLVED` — SecretRef 指向的 env 不存在
- `REF_SHADOWED` — 同一 provider 多处重复配置
- `LEGACY_RESIDUE` — 遗留明文

`pnpm openclaw secrets audit --check` 期望输出 `clean. plaintext=0, unresolved=0, shadowed=0, legacy=0`。

### 6.5 潜在劫持/攻击面分析

#### 6.5.1 本地 127.0.0.1 攻击者

| 能力 | 可行性 | 影响 |
|---|---|---|
| 暴力破解 token | ✅(loopback 速率限制豁免) | 完全控制 |
| 读 `.openclaw/openclaw.json` | ❌(0o600) | — |
| 读 `.openclaw/exec-approvals.json` | ❌(0o600) | — |
| 访问 Control UI | ✅ | 仍需 token |
| 注册恶意 channel plugin | ✅(若可写 plugin-skills 目录) | 完全控制 |

#### 6.5.2 拿到 gateway token

可调用所有 RPC:`/v1/chat/completions`、`/v1/responses`、`/tools/invoke`、读 session 历史、kill session、读 assistant media、触发 hooks。

**等于完全控制 agent**,且因 exec-approvals 默认 `full`,可执行任意命令。

#### 6.5.3 改 openclaw.json / exec-approvals.json

需先有写权限。可:

- 改 `bind: "0.0.0.0"` + `auth.mode: "none"` → 公网无认证
- 改 `controlUi.allowedOrigins: ["*"]` → CSRF 放行
- 改 exec-approvals → RCE

#### 6.5.4 公网误配 0.0.0.0

`server-runtime-config.ts:140-144` 会拒绝无 auth 的非 loopback。但若 attacker 同时拿到 token,可外网访问 Control UI、调用 RPC。配合 DNS rebinding + `dangerouslyAllowHostHeaderOriginFallback: true` 可绕过 Origin 检查。

#### 6.5.5 微信通道风险

`@tencent-weixin/openclaw-weixin` 插件代码不在仓库内。机制上 ``gateway-auth-bypass.ts:25-32`` `resolveBundledChannelGatewayAuthBypassPaths` 允许 channel plugin 注册无需 gateway auth 的 HTTP 路径(webhook 入口必需)。

**安全责任在插件本身**:必须验证微信签名、限制 sender。

**但 openclaw 的微信插件不依赖 webhook**,所以这个攻击面对它不适用。它走 long-poll,所有消息来自 iLink 服务器,不存在"伪造入站消息"的路径(除非 iLink 服务器被攻陷)。

#### 6.5.6 DNS 劫持 / 中间人

`openclaw.json:103` `baseUrl: "https://api.deepseek.com"` 是 HTTPS,但**无 cert pinning**。DNS 劫持 + CA 共谋可中间人。

openclaw 有 SSRF 防护 ``src/infra/net/ssrf.ts`` 但仅限出站 fetch,不影响 deepseek 调用。

### 6.6 Allowlist 机制

#### 6.6.1 编译与匹配

``allowlist-match.ts:35-41`` `compileAllowlist` 把 entries 转 Set。

`resolveCompiledAllowlistMatch`:

- 空集合 → `allowed: false`(拒绝)
- 含 `"*"` → `allowed: true`(全放行)

#### 6.6.2 dmPolicy

``dm-access.ts:25-29``:

```typescript
type ChannelDmPolicy = "pairing" | "allowlist" | "open" | "disabled";
```

- `dmPolicy: "open"` + `allowFrom` 不含 `*` → `ensureOpenDmPolicyAllowFromWildcard` **自动注入 `"*"`** → 全放行 ⚠️
- `dmPolicy: "allowlist"` + `allowFrom: []` → 拒绝
- `dmPolicy: "pairing"` → 走配对流程

**当前 openclaw.json 未配置任何 channel allowlist**,完全依赖微信插件自身的 sender 校验。

### 6.7 当前项目实际安全状态

| 项 | 配置 | 状态 |
|---|---|---|

| Token 来源 | `OPENCLAW_GATEWAY_TOKEN` env | ✅ 不入库 |

## 八、总结

openclaw 的通信架构可以用一句话概括:**"主动出站、loopback 监听、in-process 协作"**。

### 8.1 设计哲学

| 传统 bot 架构 | openclaw 架构 |
|---|---|
| 平台 webhook → 你的服务器 | 你的服务器 → 平台 long-poll |
| 需要公网 IP 或穿透工具 | 完全不需要 |
| 通道进程独立 | 通道 in-process |
| agent 子进程 | agent in-process |
| 端口对公网开放 | 端口仅本机访问 |

### 8.2 安全优势

1. **攻击面极小**:gateway 默认 loopback,公网完全不可达
2. **认证扎实**:`timingSafeEqual` 防时序侧信道,文件 `0o600` 防本地读,SecretRef 分离明文
3. **Symlink 防御**:`O_NOFOLLOW | O_EXCL` 防 symlink/hardlink 替换攻击
4. **去重与重试**:durable receive journal + durable delivery 保证消息不丢不重

### 8.3 待改进的点

1. **Token 不支持过期/轮换** — 一次泄露终身有效
2. **Loopback 速率限制豁免** — 本地暴力破解理论可行
3. **exec-approvals 默认 `full`** — 违反最小权限原则
4. **`assistant-media` 允许 query token** — 日志泄露面
5. **微信通道无 allowlist** — 完全依赖插件默认行为

### 8.4 给读者的建议

如果你也想用 openclaw + 微信通道搭一个 AI 员工:

1. **保持 `bind: "loopback"`** — 不要为了"方便"改成 lan
2. **走 SecretRef** — 不要图省事把 token 写进 openclaw.json
3. **配 allowlist** — 至少把 `dmPolicy` 设成 `"allowlist"` + 显式 `allowFrom`
4. **exec-approvals 改 allowlist 模式** — 给 agent 列出允许的命令
5. **不要在 Trae IDE 内置终端跑 `openclaw gateway install`** — 见 ``openclaw_use.md § 1.3`` 的避坑说明
6. **定期 `pnpm openclaw secrets audit --check`** — 确保无明文凭据

### 8.5 一句话回答用户的 5 个问题

1. **微信通道的原因是什么?** — 微信是中国市场主导聊天平台,做通道让 AI 员工触达用户日常会话入口;由于没有官方 bot API,只能登录个人微信号走 iLink long-poll
2. **是怎样实现内网穿透的?** — **不需要内网穿透**。整个通信路径全是出站 HTTPS(long-poll + CDN),gateway 主动连 iLink 服务器,服务器不需要反向连回来
3. **从微信发消息到收到回复的通信怎么做的?** — 见第五节完整时序,共 30 步,涉及 monitor → kernel(ingest/classify/preflight/resolve/dispatch)→ agent runner → LLM → deliver → 通道 sendText → iLink API
4. **网络有没有被劫持的风险?** — 有,但攻击面小。最大风险点:① 本地 token 暴力破解(loopback 速率限制豁免)② token 泄露后完全控制 agent ③ DNS 劫持 + CA 共谋中间人(无 cert pinning)④ 误配 `bind: "0.0.0.0"` 暴露公网
5. **为什么 gateway 监听本地端口就可以了?** — 因为 gateway 主动出站连接微信服务器(不需要外部连进来),微信通道在 gateway 进程内(in-process 调用不经过网络),agent 也在 gateway 进程内(同样 in-process)。本地端口只是给本机 CLI / Control UI / HTTP API 客户端用的
