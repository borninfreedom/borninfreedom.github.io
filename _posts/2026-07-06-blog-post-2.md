## 1. 背景与目标

大语言模型（LLM）推理面临两大瓶颈：显存容量与全局内存带宽。FP8 和 FP4 等低精度浮点格式能在几乎不损失模型效果的前提下，将权重和激活的大小缩小至原始 FP16 的 1/2 甚至 1/4，直接缓解显存压力。然而，仅仅缩减 bit 数并不够——为了在 GPU 上高效执行矩阵乘法，必须精心设计数据布局、解量化流程以及缩放因子的存储与访问方式。

TensorRT-LLM 没有将量化停留在“离线转换 + 简单反量化”的层面，而是把 FP4/FP8 GEMM 封装为一套可插拔的 TensorRT Plugin。这些 Plugin 内部使用 CUTLASS 生成的模板化 GEMM kernel，或者直接调用手写 CUDA kernel，并在构建期通过 profiling 自动选择最优的执行配置（tactic）。本文将深入 FP4 Block‑Scaling GEMM、FP8 Rowwise GEMM 和 Weight‑Only Groupwise Quantized Matmul 三个典型 Plugin，拆解其数据布局、缩放因子计算、向量化加载以及底层指令路径，揭示 TensorRT-LLM 如何在保证准确度的前提下榨干 GPU 的字节带宽。

## 2. 核心设计与实现

### 2.1 FP4 Block‑Scaling GEMM

FP4 量化将每个 4‑bit 数值打包为半个字节，两个 4‑bit 值共享一个字节，从而使矩阵的内存占用降为 FP16 的 1/4。但单纯的 4‑bit 变换范围极小（例如 E2M1 格式只能表示 8 个数），必须配合缩放因子来恢复动态范围。TensorRT-LLM 的做法是采用 **block‑scaling**：将输入矩阵和权重矩阵划分为大小为 `sfVecSize`（16）的连续向量块，每个块共享一个 FP8 缩放因子，最终再加一个全局 float 缩放因子。

数据布局是性能的关键。在 Plugin 初始化阶段，会计算缩放因子数组的尺寸并分配 workspace：

```cpp
// cpp/tensorrt_llm/plugins/fp4GemmPlugin/fp4GemmPlugin.cpp:48-52
int vector_size = 16;
int sf_round_m = ((m + 127) / 128) * 128;
int sf_round_n = ((n + 127) / 128) * 128;
int sf_round_k = ((k / vector_size + 3) / 4) * 4;
float* a_sf = reinterpret_cast<float*>(
    nextWorkspacePtr(wsBytePointer, wsByteOffset, sf_round_m * sf_round_k));
float* b_sf = reinterpret_cast<float*>(
    nextWorkspacePtr(wsBytePointer, wsByteOffset, sf_round_n * sf_round_k));
float* global_sf = reinterpret_cast<float*>(
    nextWorkspacePtr(wsBytePointer, wsByteOffset, sizeof(float)));
```

这里 M 和 N 被向上对齐到 128 的整数倍，K 则在按 vector_size 分组后再对齐到 4。对齐后的缩放因子数组每一行恰好 128 字节——正好对应一条 GPU 的 128‑bit 合并访问事务。这意味着一个 warp 可以用单条 `ld.global.v4.b32` 指令加载一整行的 scale，有效避免了 bank conflict，并为后续的广播与乘加提供完美的向量化模式。

实际的 GEMM 调用委托给模板化的 CUTLASS Runner：

```cpp
// cpp/tensorrt_llm/plugins/fp4GemmPlugin/fp4GemmPlugin.cpp:82-93
if (mOutputType == nvinfer1::DataType::kFLOAT)
{
    mGemmRunner = std::make_shared<tensorrt_llm::kernels::CutlassFp4GemmRunner<float>>();
}
else if (mOutputType == nvinfer1::DataType::kHALF)
{
    mGemmRunner = std::make_shared<tensorrt_llm::kernels::CutlassFp4GemmRunner<half>>();
}
else if (mOutputType == nvinfer1::DataType::kBF16)
{
    mGemmRunner = std::make_shared<tensorrt_llm::kernels::CutlassFp4GemmRunner<__nv_bfloat16>>();
}
```

模板参数 `OutputType` 决定了中间累加器类型以及最终写回的精度，在编译期即生成三种完全特化的 kernel，避免了运行时分支。Kernel 内部，每个线程块加载自己的 tile 的 FP4 数据，通过硬件支持的 `cvt` 指令（Pascal 以上）或位操作将其转换为 FP16 中间表示，乘以本地的 scale（已预先广播至寄存器），再送入 Tensor Core 的 `mma.sync` 指令完成乘累加。最后 global scale 作用于整个输出矩阵，提供粗粒度的动态范围调整。

### 2.2 FP8 Rowwise GEMM

FP8 格式（E4M3 或 E5M2）是 NVIDIA Hopper 架构原生支持的数据类型。TensorRT-LLM 的 `Fp8RowwiseGemmPlugin` 进一步增强了这一能力：激活矩阵的每一行（per‑token）和权重矩阵的每一列（per‑channel）都可以附带独立的 float 缩放因子。这种方式比 per‑tensor 有更高的精度保留度，而相比 block‑scaling，它减少了 scale 数据的体积（只需 `(m + n)` 个 float，而不是 `(m*k/block_size + n*k/block_size)`）。

运行时，Plugin 的 `enqueue` 将输入指针分别映射为输入、权重和两个缩放因子数组，然后调用 CutlassFp8GemmRunner：

```cpp
// cpp/tensorrt_llm/plugins/fp8RowwiseGemmPlugin/fp8RowwiseGemmPlugin.cpp:175-182
mGemmRunner->gemm(outputs[0], inputs[0], inputs[1], nullptr, mQuantMode,
    m, n, k,
    reinterpret_cast<float const*>(inputs[2]),        // per-token scale
    reinterpret_cast<float const*>(inputs[3]),        // per-channel scale
    *bestTactic,
    reinterpret_cast<char*>(workspace), wsSize, stream);
```

在 CUTLASS kernel 内，每个线程块被分配 M 方向的一行 tile 和 N 方向的一列 tile，它会从全局内存加载对应的 scale 值到共享内存。在计算 `C[i][j] += A[i][k] * B[k][j]` 之前，先执行 `A_fp16 = float(A_fp8) * scaleA[i]` 和 `B_fp16 = float(B_fp8) * scaleB[j]`，乘积结果送入 Tensor Core。因为 scale 被广播至 shared memory，所有 warp 可近乎零延迟地获取。这种“先提升精度再运算”的模式虽然在逻辑上多了一步转换，但在 Hopper 的 FP8 Tensor Core 中已硬件支持，实际只消耗一条 `mma.sync` 指令。

### 2.3 Weight‑Only Groupwise INT4/INT8 GEMM

在很多部署场景中，激活保持 FP16 或 BF16，仅对权重进行低位量化，这便是 Weight‑Only 量化。TensorRT-LLM 的 `WeightOnlyGroupwiseQuantMatmulPlugin` 支持 INT4 和 INT8 两种打包方式，并引入 **groupwise** 缩放与零点：每 `group_size`（默认 128）个权重共享一个 scale 和一个零点。

权重的存储方式为：INT4 将 4 个 4‑bit 值打包为一个 `int32_t`（即 16‑bit 的 half‑word），INT8 则将 2 个 8‑bit 值打包为一个 `int32_t`。如下代码展示了在 CUTLASS 接口中，`int32_t*` 指针如何指向打包的权重：

```cpp
// cpp/tensorrt_llm/plugins/weightOnlyGroupwiseQuantMatmulPlugin/weightOnlyGroupwiseQuantMatmulPlugin.cpp:288-302
int real_n = mQuantAlgo & GroupwiseQuantAlgo::INT8_WEIGHT
                  ? n * FP16_INT8_RATIO   // 2
                  : n * FP16_INT4_RATIO;  // 4，每个 int32 包含 4 个 INT4

if (use_cuda_kernel)
{
    // 使用手写 CUDA kernel for small batch
    tensorrt_llm::kernels::weight_only::Params params{...};
    tensorrt_llm::kernels::weight_only::kernel_launcher(mArch, params, stream);
}
else
{
    // 使用 CUTLASS kernel for large batch
    mWeightOnlyGemmRunner->gemm(...);
}
```

Plugin 会根据 `m`（batch 维度）的大小自动选择 kernel：当 `m < 16` 时，使用专为极小 batch 优化的手写 CUDA kernel，减少 launch 延迟和共享内存浪费；否则走 CUTLASS 通用路径，用 tile‑based 策略淹没延迟。

解量化的方程是 `w_deq = (w_int - zero) * scale`。在 CUTLASS kernel 中，预取阶段会将属于当前 tile 的 scale 和零点从全局内存加载到共享内存，后续每个线程在从打包的 `int32_t` 中提取出多个低精度整数时，可以直接从共享内存读取对应的 scale/zero 进行解量化。接着，如果启用了 FP8 Alpha 模式，激活也会在 GEMM 前与 per‑channel 的预量化 scale 相乘，然后截断为 FP8。这两种技术（groupwise 解量化权重 + 激活的预 scale）融合在一枚 kernel 中，做到了 W4A8 的近似效果，利用 FP8 Tensor Core 的高吞吐，同时避免全 FP8 训练所需的校准成本。

## 3. 优化技术细节

### 3.1 内存对齐与向量化加载

FP4 Plugin 中对 M 和 N 的 128 字节对齐，以及 K 方向按 `vector_size=16` 对齐到 4，是最核心的访存优化。当 scale 数组的行大小为 128 字节时，GPU 的合并访问单元可以一次性服务 32 个线程（每个线程 4 字节），没有 bank conflict。这同样适用于 scales 的广播：在 SM 内部，scale 行被加载到共享内存后，各个 warp 以相同步长重复使用，实现极低的重复加载开销。

### 3.2 模板化 Runner 与 Profiling 协同

三个 Plugin 均使用了模板化的 CUTLASS Runner（`CutlassFp4GemmRunner`、`CutlassFp8GemmRunner` 等）。这些 Runner 在构建期会遍历所有可能的 tile 大小、warp tiling 方式和指令选择（tactic）。例如，FP4 GEMM 的 profiling 会真正分配模拟的输入/输出和 workspace，然后调用 `runner->gemm`。如果某个 tactic 需要的共享内存超过 SM 限制，CUTLASS 会在 `gemm` 内部抛出 C++ 异常，Plugin 捕获后将其标记为无效，确保最终留在 tactic 池里的都是合法配置。这种“以跑代检”的方式虽然增加了 engine 构建耗时，但完全消除了运行时的不确定性。

### 3.3 动态 Kernel 选择

Weight‑Only Plugin 同时注册 CUDA 手写 kernel 和 CUTLASS kernel。在 profiling 阶段，两种 kernel 根据传入的 `m` 分别计时，最终记录为两个独立的 tactic。运行时，Plugin 根据实际 `m` 的大小选择更优的哪一个。手写 kernel 常针对低 batch 使用 `dp4a`（定点 4 路点积）指令，无需复杂的 tile 划分，且各 warp 间无需同步便能完成全矩阵，能够显著减少小 batch 场景下的 tail effect。

### 3.4 Pre‑Quant Scale 融合

在 Weight‑Only Plugin 的 W4A8 模式下，激活首先乘以一个 per‑channel 的 `pre_quant_scale`，然后截断为 FP8。该操作融合在 Weight‑Only Groupwise GEMM kernel 的主循环中，而不是作为单独的前置 kernel 执行。代码中，当检测到 `use_pre_quant_scale` 标志，便从额外输入索引读取 scale，并在每个 tile 加载激活时就地完成缩放与类型转换。这样一来，激活从 HBM 读取的次数减少一次，且避免了中间 tensor 的写回与重新读取。

## 4. 性能与权衡

上述优化的共同目标是在保证模型精度的前提下最大化 GPU 的内存带宽利用率。

- **内存占用**：FP4 将权重和激活压至 FP16 的 1/4，但缩放因子会带来额外开销。对于一个大矩阵，block‑scaling 的 scale 开销约占原始数据的 1/(16*2) = 3.125%（scale 为 FP8），整体净内存节省仍非常可观。Weight‑Only 的 groupwise 模式中，scale 和零点的开销为 `(k / group_size) * n * 2` 字节（half 存储），当 `group_size=128` 时额外约 3.1%。
- **计算吞吐**：FP8 在 Hopper GPU 上的 Tensor Core 吞吐可达 FP16 的 2 倍，FP4 则通过硬件转换后利用 FP16 或 TF32 的 Tensor Core，实际吞吐受转换开销影响，但依旧远高于逐 bit 解量化。从 NVIDIA 公开数据来看，FP4 GEMM 的 peak throughput 约为 FP8 的 50‑70%，取决于具体 tile 尺寸。
- **精度损失**：Block‑scaling 和 groupwise 量化都依赖校准数据来决定 scale。col-wise 或 row‑wise 的 scale 可很好地保留 outlier 信息，但与细粒度 block 相比，某些 token 内的分布差异可能无法完全捕获。TensorRT-LLM 允许用户在 Python 侧通过量化工具箱（`quantize.py`）进行加权校准或选择不同的 group size，以平衡精度与带宽。工程实践中，对于 Llama、Mixtral 等模型，PTQ 后的困惑度劣化通常在 0.1 以内。
- **工程代价**：模板化 Runner 导致编译时间增加，二进制体积膨胀（每个量化组合生成一份 kernel）。但这是编译期成本，运行时无影响。Profiling 构建时间从几秒到数十秒不等，对于生产部署可接受。

## 5. 相关论文

- **FP8 Formats for Deep Learning** (Micikevicius et al., 2022)：提出 E4M3 与 E5M2 两种 FP8 格式，并讨论其训练与推理中的用法。TensorRT-LLM 的 FP8 量化遵循该规范，文件引用见 `cpp/tensorrt_llm/quantization/fp8` 与 `tensorrt_llm/quantization/mode.py`。
- 其他可能相关的量化论文未被直接引用在 Plugin 代码中，但 Block‑Scaling 概念受 NVIDIA 稀疏矩阵与微缩放格式（如 MXFP4/MXFP8）的启发。

## 6. 总结

TensorRT-LLM 的量化 GEMM Plugin 并不是简单地调用 `cublasLt` 的低精度接口，而是从数据布局到 kernel 调度做了全栈定制。FP4 Block‑Scaling 通过对齐 padding 实现完美的 Scale 合并访问；FP8 Rowwise GEMM 以最低的额外数据量保留了 per‑token/per‑channel 的精度；Weight‑Only Groupwise 则结合打包与在线解量化，用细粒度 scale 弥补低位宽的精度损失。模板化的 CUTLASS Runner 和 profiling 机制将战术选择下沉到构建期，确保每一种量化配置都能在目标 GPU 上以最优方式执行。这些工程实现在不牺牲准确率的前提下，让 FP4/FP8 的带宽红利真正转化为推理吞吐的提升。
